## 0. 한 줄 핵심

> **"Claude를 잘 쓴다"는 건 모델이 알아서 완벽하길 바라는 게 아니라,
> 실수를 자동으로 걸러주는 안전장치를 깔아서 더 많이 믿고 맡기는 것.**

검증을 *사람의 기억*이 아니라 *시스템*에 박는다 — 이게 핵심 원칙.

---

## 1. As-Is → To-Be

| | As-Is (처음) | To-Be (목표) |
|---|---|---|
| 검증 | 사람이 기억해서 테스트 실행 → 까먹음 | 시스템이 자동 검사 |
| 코드 리뷰 | 하기로 해놓고 건너뜀 | PR마다 봇이 자동 리뷰 |
| 실수 | 사람 눈에만 의존 | 자동으로 1차 필터링 |
| 위임 | "코드 짜줘" 식 단발 | 안전망 위에서 크게 위임 |
| 비유 | 안전벨트 없이 조심운전 | 안전벨트·자동브레이크 깔린 차 |

---

## 2. 핵심 부품 용어집 (비유)

| 용어 | 한 줄 비유 | 이 프로젝트에서 |
|------|-----------|------------------|
| **CI** | 공장 끝의 자동 검수대 | `ci.yml` — PR마다 문법/타입/테스트 자동 검사, 틀리면 빨간불 |
| **리뷰 봇** | 자동 첨삭 선생님 | `claude-code-review.yml` — PR마다 AI가 코드 읽고 댓글 |
| **훅(hook)** | 자동문 센서 / 도어락 | 특정 순간 자동 실행 (예: 커밋 직전 테스트, main 커밋 차단) — 아직 미구현 |
| **권한 allowlist** | 경비실 프리패스 명단 | 안 묻고 실행할 명령 목록. 위험한 건 매번 확인(`ask`) |
| **PR** | "이거 합쳐도 돼요?" 제안서 | 검사기·봇이 여기 붙어서 검토 |
| **머지(merge)** | 제안서를 본진(main)에 합침 | 검사 통과 후 합류 |
| **dogfooding** | 사장이 자기 음식 먼저 먹어보기 | 만든 봇을 우리 코드에 직접 돌려 설정 결함 발견·수정 |
| **DoD** | "이거 끝났다"의 합격 기준 체크리스트 | 구현→테스트→검증→리뷰→머지 |

**결정론 vs 비결정론** (중요): CI 검사(기계: 자·저울)는 같은 코드 → 항상 같은 결과.
리뷰 봇(사람 같은 AI)은 매번 의견이 조금씩 다름. 그래서 봇 "통과"는 *사실*이 아니라 *조언*.

## 3. 설계 결정 Q&A (왜 이렇게 했나)

> 작업 중 실제로 부딪힌 질문과 그 결정 근거. 각 항목 = 질문 → 핵심 답 → 왜 중요.

### Q1. CI 게이트는 GitHub 안전장치일 뿐인데, 왜 "Claude 활용도"를 높이나?
- **답**: CI는 *Claude의 비결정성*에 강건해지는 장치. "검증을 기억하라"를 "시스템이 검증한다"로 바꿈.
- **왜 중요**: AI에 많이 위임할수록(손으로 덜 볼수록) 자동 안전망의 가치가 커짐. CI가 있어야 *안심하고 더 위임* → 그게 활용도.

### Q2. 봇이 "pass"라고 하면 그냥 통과인가?
- **답**: 아니. 봇 pass = "봇이 돌았고 코멘트 없음"(약한 신호)이지 "버그 없음 보증"이 아님.
- **왜 중요**: 진짜 게이트는 *결정론적 CI(test/typecheck)* + 사람. 봇은 두 번째 눈이지 권위가 아님.

### Q3. 봇도 CI의 일부 아닌가?
- **답**: 넓게 보면 맞음(같은 GitHub Actions, PR마다 실행). 단 *성격*이 다름 — CI 검사 = 기계(결정론·게이트), 봇 = AI(조언).
- **왜 중요**: "같은 인프라 ≠ 같은 신뢰도". 자동화라고 다 동급으로 믿으면 안 됨.

### Q4. 훅으로 검사하는 거랑 CI로 검사하는 거 겹치지 않나?
- **답**: 테스트 훅은 CI와 *겹침* — 차이는 "언제/어디서"(훅=로컬·즉시, CI=원격·나중). 훅은 빠른 조기경보, CI는 못 건너뛰는 최종 게이트. ("shift-left")
- **반전**: 그래서 *겹치는 훅(테스트)*은 속도만 더해 가치가 낮고, *안 겹치는 훅(main 커밋 차단)*이 진짜 가치 — CI는 커밋 *후*라 커밋 자체를 못 막음.

### Q5. 권한 allowlist 큐레이션이 뭔가?
- **답**: 세션 중 박제된 일회성 권한 21개를 지우고, 안전·빈번한 건 패턴으로 압축(자동), 위험한 변형(force push 등)은 `ask`로 매번 확인.
- **왜 중요**: 우선순위 `deny > ask > allow` → `git push *`는 자동이되 `git push --force*`만 가로챔. "편의 ↑ + 안전 유지"의 균형.

### Q6. (메타) 왜 이런 자동화에 공을 들이나?
- **답**: 화려한 프롬프트보다 *규율 + 자동 안전망*이 결과 차이의 대부분. 좋은 엔지니어링 규율(작은 PR·테스트 우선·리뷰)을 그대로 AI 협업에 입히는 것.

## 4. 진행 타임라인 (무엇을 했나)

1. **진단** — Claude 활용도 B+ (규율은 좋은데 자동 안전장치 없음).
2. **CI 게이트(T1)** — PR마다 typecheck/test/lint 자동 검사 추가.
3. **리뷰 봇(T3)** — PR마다 AI 자동 리뷰. *직접 돌려보다(dogfooding)* 비용캡(`--max-turns 10`)이 리뷰를 통째로 망가뜨리던 걸 발견 → 40으로 수정, 리뷰 1회 ≈ $1.20 실측.
4. **권한 정리(T5)** — allowlist 21개 → 17 패턴 + 7 ask 가드.
5. **실제 기능** — 감정 분석(데이터 계층 + AI 엔진)을 새 안전망 위에서 구현·머지.
6. **남은 것** — 검증 훅(T2, 커밋 직전 자동 검사 + main 커밋 차단), CLAUDE.md 보강(T4), 서브에이전트 교차검증(T6).

**지금 위치**: 큰 안전장치(자동 검사 + 자동 리뷰 + 권한 정리)는 완료, 남은 건 더 촘촘하게.

---

## 5. 요약 (한 문장)

> AI 코딩을 잘 쓰는 핵심은 *모델이 완벽하길 기대*하는 게 아니라, *실수를 자동으로 잡는 안전망(CI·리뷰봇·훅)을 깔아 더 많이 위임*하는 것 —
> 리뷰 봇 도입 시 직접 돌려보며(dogfooding) 비용 설정 결함을 잡아 고쳤고, '봇 통과'는 결정론적 CI나 사람 판단과 달리 *조언*일 뿐이라는 신뢰도 구분을 두고 일했다.
